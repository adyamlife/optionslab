hard filter pass — StructureConstraints
✓Pre-Scoring Hard Filters — _hard_filter_pass() LIVE
Binary pass/fail gates checked before signal scoring. A structure that fails any gate is eliminated entirely and never enters the scoring engine. Rules live in each structure's OptionStructure.constraints: StructureConstraints field (config/structures/_base.py). Failures surface in output as hard_filter_hits[] and rejection_reason.
StructureConstraints dataclass fields
min_dte / max_dte — DTE range gate
earnings_dte_min — reject if earnings < N days (credit: too close)
earnings_dte_max — reject if earnings > N days (vol: too far / absent)
min_iv_rank / max_iv_rank — IV rank bounds
allowed_trends — empty = all pass
Replaces the old earnings_buffer field with an explicit symmetric pair: credit structures use earnings_dte_min (earnings too close = event risk); vol-expansion structures use earnings_dte_max (earnings too far = no catalyst). Unknown earnings date with earnings_dte_max set also triggers rejection.

reject ifearnings_dte unknown
earnings_dte_max=21 — thesis is the event itself; no confirmed catalyst = no trade
Long Strangle LIVE
reject ifearnings_dte > 25d
reject ifearnings_dte unknown

Iron Butterfly LIVE
reject ifearnings_dte < 10d

Called once per candidate  ·  weights + preferences from structure_scores.toml  ·  thresholds from signal_params.toml  ·  25 registered evaluators
Generic evaluate() dispatch
Each signal has a registered evaluator function. The engine dispatches by signal name:
evaluate(signal, preference, market) → EvaluationResult
EvaluationResult
score      float ∈ [-1, +1] or None
confidence float ∈ [0, 1]

score = None when…
required market field absent (stale / None)
→ signal skipped entirely
→ excluded from effective_max
→ if required=true: required_missing[] + ×0.70 penalty per signal
required = true — distinguishes missing data from unfavorable data LIVE
A signal marked required = true in structure_scores.toml has a different failure mode than an optional signal:
— Optional signal absent: silently excluded from effective_max. Score proceeds with remaining signals.
— Required signal absent: added to required_missing[]; confidence_multiplier ×= 0.70 per missing required signal.
Two missing required signals → ×0.49 multiplier applied to the final pct before rating and output.

Unusual OI Activity NEW unusual_activity detected | quiet Max per-strike OI spike ratio vs 10-day rolling avg (±10% of spot). 0 = quiet; 1.0 = OI doubled; capped at 5×. detected ≥1.0: @strong for Long Straddle/Strangle + directional. quiet <0.25: @strong for IC/IBF/credit. Returns None until 3 days of oi_changes history accumulate. Max score 0.70.
IV/HV Ratio NEW iv_hv_ratio high | low | moderate ATM IV ÷ 20d HV. Orthogonal to iv_premium (absolute spread): ratio adjusts for base vol level. high (≥1.20): @strong for credit/premium-selling. low (≤0.85): @strong for debit/long-vol. moderate (near 1.0 ±0.10): @strong for balanced structures. Max score 0.75.
Volume PCR NEW vol_pcr bullish | bearish | neutral Put volume ÷ call volume (today's activity — fast/tactical). Orthogonal to OI PCR (slow/structural). bullish (≤0.75): call volume dominant. bearish (≥1.30): put volume dominant. neutral (0.95–1.05): balanced flow. Max score 0.65.
PCR Divergence NEW pcr_diverge unwinding_bears | building_bears | neutral oi_pcr − vol_pcr (signed). Positive = OI bearish but vol bullish (bears unwinding / bulls entering). Negative = OI bullish but vol bearish (new put buying). |diverge| ≥ 0.30: significant structural shift. unwinding_bears: @strong for bullish structures. Max score 0.60.
eff_wt = base_weight × regime_mult
|
contribution = eff_wt × result.score × result.confidence
|
effective_max = Σ eff_wt for non-None signals
|
raw_pct = score ÷ effective_max
|
pct = raw_pct × confidence_multiplier (0.70ⁿ per required_missing)
Confidence — data quality / decisiveness, independent of score direction
1.00 — unambiguous; all data present, decisive
0.70 — good signal; minor uncertainty
0.50 — transitional zone; real but weak
0.35 — fallback path (e.g. adx_slope missing, proxy used)
None — required data absent → score=None → skipped
0.70ⁿ — confidence_multiplier penalty per required_missing signal

FFeature Store — Persistence + Versioning LIVE
Two new DuckDB tables written every scan. feature_metadata is the feature registry (seeded with 28 features); feature_snapshots is the per-scan historical record. Together they enable version-safe training data filtering and future DAG-based staleness propagation.
feature_metadata — registry

source_type — options | yfinance | fred | derived

depends_on — JSON array of upstream feature names (DAG edges)

deprecated_at — NULL = current version; set when calculation changes
28 features seeded at first scan. Version boundary: training pipelines filter by version so rows produced before a calculation change stay labeled v1; rows after as v2.
